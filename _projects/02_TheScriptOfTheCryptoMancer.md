# Tool - Lateralus
Often times cryptographic algorithms handle large numbers and usually perform arithmetic operations in a certain algebraic groups (e.g. modulo a given positive prime number). In this problem, a variant of Diffie-Hellman key exchange protocol based on the Fibonacci sequence. Both the public keys and the secret keys of the two parties are Fibonacci numbers. Briefly, given their Fibonacci number *pk_B* in the chosen group and our Fibonacci number *sk_A*, the *sk_A-th* Fibonacci number after *pk_B* is the derived key.
Since computations are done in a group, the numbers can only grow so large. With this in mind, computing all Fibonacci numbers until we get a match with *pk_B* and computing the next *sk_A* numbers in the sequence seems like a feasible solution. It turns out not to be the case - this approach consumed a significant amount of memory and my patience ended before the algorithm did. A little bit of googling uncovered a convenient way of representing the Fibonacci sequence as a matrix and using matrix exponentiation to produce the numbers. This was great news because there are [optimized matrix exponentiation](https://www.geeksforgeeks.org/matrix-exponentiation/){:target="_blank"} algorithms which have better performance than regular multiplication.

# War Ciphers
If nothing else, wars seem to cause humanity to up its crypto game. One of the most widely known encryption device was the Enigma used in World War II ???? There seems to have been plenty of other encryption schemes used for secret communication during war time such as the **[WW II Slovak Columnar Transposition Cipher](https://ep.liu.se/ecp/article.asp?issue=171%26article=4){:target="_blank"}** and the **[WW I ADFGVX German Cipher](https://en.wikipedia.org/wiki/ADFGVX_cipher){:target="_blank"}**. The definition of both is quite notation-heavy, so I will refer you to the relevant pdf in the repository, if interested. Instead, I will provide a more informal overview of the two ciphers. <br/>

The columnar transposition is defined by a key which is a permutation of all integers up to a certain number **k** (e.g. the number of columns to be used). The message to be encrypted is written in a table with **k** columns, one character per cell, wrapping around to a new row every **k** characters. Padding is used to fill up the last row. Then, the columns are read in a top-down fashion, in order according to the permutation and voila - we have our ciphertext. Below you can see as an example the plaintext *lake town* encrypted to the ciphertext *"at?ew?l_nko?"* given the permutation *(3, 1, 4, 2)* and for *k = 4*. <br/>
<p align="center"><img width="25%" src="/images/Crypto/ColumnarTransposition.JPG"></p>

For the **ADFGVX German Cipher**, I will only mention it's basic building blocks. The secret key of the algorithm is defined by an invertible substitution function called a *[Polybius Square](https://en.wikipedia.org/wiki/Polybius_square){:target="_blank"}* (often represented by a matrix which has the supported alphabet for entries), a *padding character*, and a *permutation*. A given plaintext is first split into *n* substrings and the substitution function is performed on each of them concatenating all of the outputs. To obtain the final ciphertext, the columnar transposition as seen above is applied to this intermediary result using the specified padding character and permutation.

# PQR, a.k.a. triple RSA
[RSA](https://www.di-mgt.com.au/rsa_alg.html){:target="_blank"} is a widely used public-key cryptosystem with computations performed modulus an integer which is the product of two large prime numbers, e.g. *N=p\*q*. The modulus is used for encryption and is made public, the security of RSA relies on the practical difficulty of the factoring problem (e.g. find the prime factorization of an integer) with respect to the modulus. <br/>
<p align="center"><img width="75%" src="/images/Crypto/RSA.JPG"></p>

In this problem, the RSA cryptosystem has modulus of the form *N=p\*q\*r*, e.g. *N* is the product of three prime numbers. We make use of faults in the cryptosystem to recover two of the prime numbers and simply compute the third. One characteristic of the cryptosystem is that moduli are generated using a low entropy random seed. In other words, there is a high probability of generating the same prime number twice and subsequently use it to compute two different moduli. Given access to enough moduli from this cryptosystem, we can compute the *greatest common divisor* between every pair. If the GCD is prime and different from 1, then we have found our first prime divisor. <br/>

We also know that the cryptosystem endures some occasional errors during the decryption process - in particular, during the exponentiation of the ciphertext modulo one of the prime numbers. This produces a corrupt decrypted message. In addition, decryption performance is improved using the *[Chinese Remainder Theorem](https://en.wikipedia.org/wiki/RSA_(cryptosystem)#Using_the_Chinese_remainder_algorithm){:target="_blank"}*. With a pair of plaintext and the corresponding corrupt decrypted message we can perform a fault-attack which readily reveals the prime number modulo which the fault occurred. You can read more on [fault attacks on CRT-based RSA](https://eprint.iacr.org/2002/073.pdf){:target="_blank"}, as well as a more detailed [step-by-step attack](https://www.cryptologie.net/article/371/fault-attacks-on-rsas-signatures/){:target="_blank"}. <br/>

Knowing the factorization of the modulus and the public exponent (refer to the RSA algorithm specifications for details), we can recover the secret *d* by applying the [Extended Euclidean Algorithm](https://en.wikipedia.org/wiki/Extended_Euclidean_algorithm){:target="_blank"} for modular inversion. And we are done! We have recovered the secret key, so now we can freely decrypt anything that has been encrypted with the corresponding public key!

# Mr. Copper & Ms. Smith
In order to reduce encryption or signature-verification time, RSA-based cryptosystems often use certain small public exponent *e* which have been proven to significantly optimize modular exponentiation. Besides computation time reduction, this can also bring some security risks. While attacks based on small RSA public exponent do not completely break the cryptosystem (e.g. they do not usually lead to recovering the secret exponent *d*), they can still be applied to recover certain messages. The attacks relevant to this problem are quite notation-heavy so, for the time being, I will only leave a few links for you to explore - a detailed paper on [various attacks on RSA](http://crypto.stanford.edu/~dabo/pubs/papers/RSA-survey.pdf){:target="_blank"} with the most relevant ones being those based on the [Coppersmith's Method](https://en.wikipedia.org/wiki/Coppersmith%27s_attack){:target="_blank"}. # You'll Never (Random) Walk Alone
Here we deal with *[commitment schemes](https://en.wikipedia.org/wiki/Commitment_scheme){:target="_blank"}* - a cryptographic primitive which allows for the sender to commit to a given value and at the same time keep it hidden from other parties. At a later stage, this value can be revealed and confirmed. A commitment scheme has to provide two properties - the **hiding property** guarantees that the committed value remains unknown to all but the sender until the reveal phase of the scheme; and the **binding property** guarantees that the only value which validates during the reveal phase is the real value originally committed to by the sender. If we are not careful when defining a commitment scheme, an adversary can devise an attack which breaks one or both of these properties. <br/>

In the first part of this problem we show how to perform a forgery attack against the *[Pedersen Commitment Scheme](https://asecuritysite.com/encryption/ped){:target="_blank"}* (e.g. breaking the *binding* property). For example, the commitment value *c* in our case is computed for a given value *x* and a random *r* by feeding them to some function: **c = Commit(x, r)**. To break the *binding* property for *c* and some other value *y*, we need to find an *r_forge* such that **Commit(y, r_forge) = Commit(x, r) = c**. In other words, even if we have already committed to a value *x* and we later on realize that it is more advantageous for us to use a value *y*, we can make it look like our initial value was, indeed, *y*. In particular, we are dealing with a *rock-paper-scissors* tournament. In this context, we can commit to *rock*, but once we reveal our opponent's choice and see that it was, for example, *paper*, we can find and *r_forge* which would result in a valid reveal of *scissors* as our original value. <br/>

As an improvement, we are presented with a commitment scheme based on a hash function (see details in the relevant pdf file on GitHub). Once again, we want to perform forgery in order to break the *binding* property. This time, we will use the hash function as an attack vector, e.g. look for a collision. There is always the option to try and bruteforce your way through a problem, but more often than not there are better alternatives. [Floyd's Tortoise and Hare](https://en.wikipedia.org/wiki/Cycle_detection#Floyd's_tortoise_and_hare){:target="_blank"} cycle detection algorithm is a simple algorithm which uses two pointers to traverse a chain in one direction (e.g. a chain of hashes) in a linked-list fashion. Both pointers start from the beginning (e.g. at value *tortoise = hare = x_0* in the example below) but move differently - the *tortoise* moves one step every time, *tortoise = hash(tortoise)*, while the *hare* moves two steps every time, *hare = hash(hash(hare))*. If a cycle exists, the two are guaranteed to meet somewhere inside the cycle (at *x_16* in the image below). Once that happens, reset one of the pointers to the beginning, *x_0*, while leaving the second pointer at the meeting node. Then, advance both pointers in the same rate of one step at a time. It can be proven that the tortoise and the hare will now meet at the *collision node* (*x_5* in the image below) revealing a collision (in this case, *hash(x_4) = hash(x_20)*). Here is a short [video](https://www.youtube.com/watch?v=9YTjXqqJEFE&ab_channel=JomaClass){:target="_blank"} explaining why the algorithm actually works.
<p align="center"><img width="75%" src="/images/Crypto/Floyd.JPG"></p>
Given the way this commitment scheme is constructed, we can use a collision of the hash function to perform a forgery attack such that we can commit to a value *x* but create a valid opening for value *y*, where *hash(x) = hash(y)*.

# Tongue Twister
This one is again a very notation-heavy problem worthy of a separate post, so I will just give an overview and some relevant resources. [The Mersenne Twister](https://en.wikipedia.org/wiki/Mersenne_Twister){:target="_blank"} is the most widely used *pseudorandom number generator* and is the default one in numerous systems such as Python, Maple, MATLAB, SageMath, R, Ruby, and more. There are a few core components behind the algorithm:
 * Matrix recurrence relation - a simple [recurrence relation](https://en.wikipedia.org/wiki/Recurrence_relation){:target="_blank"} is used to generate a sequence of numbers
 * An invertible [tempering matrix](https://en.wikipedia.org/wiki/Tempered_representation){:target="_blank"} over a binary field
 * A twisted generalised [linear-feedback shift register](https://en.wikipedia.org/wiki/Linear-feedback_shift_register){:target="_blank"} of rational normal form - more generally, a type of shift register whose input bit is a linear function of its previous state <br/>

 The Mersenne Twister is parametrized by public algorithmic constants, usually chosen to satisfy certain conditions. The algorithm is first seeded using some of the constants. Then, in order to get the *N-th* random number the state is updated *N* times performing some *tempering* and, under certain conditions, a *twist transformation*. After this bulk of terminology and lack of details, the key insight is that, given sufficiently many previous random output numbers, it is possible to predict future once.
